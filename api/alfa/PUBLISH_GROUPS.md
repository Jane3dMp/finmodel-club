# Публикация групп модель → AlfaCRM (этап 3) — дизайн и основа

Направление: **модель → CRM**. Из панели одной кнопкой создавать в AlfaCRM группу,
её расписание и состав (+ при необходимости абонементы). Даты учебного года:
**старт 02.09.2026, окончание 31.05.2027**.

> Статус: подготовлена **основа** (серверные эндпоинты + справочники + сухой прогон).
> Живая публикация **не включена** — ждём утверждённое расписание и сверку id-справочников.

## Точные поля API Alfa (из исходников, проверено)

Все — `POST /v2api/{branch}/{объект}/create`, заголовок `X-ALFACRM-TOKEN`.

### group (группа)
| поле | что | из модели |
|---|---|---|
| `name` | название группы | `l.group` (поле «Группа» занятия) |
| `branch_ids` | `[int]` филиалы | `[branch]` из config |
| `teacher_ids` | `[int]` педагоги | по имени `l.teacher` → id (нужен справочник педагогов) |
| `level_id` | уровень | ⚠️ уточнить (может быть обязателен) |
| `status_id` | статус группы | ⚠️ уточнить (напр. «активная») |
| `limit` | вместимость | `l.students` (или groupSize курса) |
| `note` | заметка | курс/примечание |
| `b_date`,`e_date` | период | 2026-09-02 … 2027-05-31 |

### regular-lesson (регулярное расписание группы)
| поле | что | из модели |
|---|---|---|
| `related_class` | `"Group"` | константа |
| `related_id` | id созданной группы | из ответа group/create |
| `subject_id` | предмет | `l.course` → subject_id (справочник subject) |
| `teacher_ids` | `[int]` | как у группы |
| `room_id` | кабинет | `l.room` → room_id (справочник room) |
| `day` | день недели | `l.day` ⚠️ сверить нумерацию Alfa (у нас 1=Пн..7=Вс) |
| `time_from_v`,`time_to_v` | время | `l.start`/`l.end` в формате `HH:MM:00` |
| `lesson_type_id` | тип занятия | справочник lesson-type (обычное занятие) |
| `b_date`,`e_date` | период | как у группы |
| `is_public` | показывать | true |

Группа 2×/нед = **две** regular-lesson (или одна с `days:[..]`).

### cgi (ученик ↔ группа)
| поле | из модели |
|---|---|
| `customer_id` | `kid.alfaId` (уже проставлен синхронизацией контактов) |
| `group_id` | id созданной группы |
| `b_date`,`e_date` | период учебного года |

### tariff / customer-tariff (абонементы — опционально, позже)
`tariff`: type, name, price, lesson_count, duration, branch_ids.
`customer-tariff`: customer_id, tariff_id, subject_ids, b_date, e_date, balance…

## Что уже готово на сервере (в этом прокси)
- `?action=refs` — тянет справочники Alfa для маппинга: **subjects, rooms, lesson-types**
  (и пытается получить педагогов/статусы/уровни; что вернётся — то и покажем).
- `?action=publish` (POST, **по умолчанию dryRun=true**) — принимает план одной группы
  (`{group, schedule[], studentAlfaIds[]}`) и:
  - при `dryRun:true` — возвращает готовые payload'ы, **ничего не создавая** (для проверки);
  - при `dryRun:false` — создаёт группу → расписание → привязку учеников, возвращает id.

## Что нужно сверить перед живым запуском (открытые вопросы)
1. **Справочник педагогов** — как в Alfa получить teacher_id по имени (эндпоинт педагогов
   в v2 не документирован явно; `refs` попробует несколько путей). Возможно, ведём таблицу
   соответствия «имя в модели → teacher_id».
2. **`status_id` / `level_id` группы** — обязательны ли и какие значения у вас в Alfa
   (узнаем сухим создdanием: Alfa вернёт ошибку валидации с требуемыми полями).
3. **Нумерация `day`** в regular-lesson (1=Пн или 0=Вс) — сверить на одном занятии.
4. **Соответствие названий курсов ↔ subjects** — если названия расходятся, делаем таблицу
   маппинга (курс модели → subject_id).

## Безопасный порядок запуска (когда расписание утверждено)
1. `refs` → получить subject/room/lesson-type/teacher id, собрать таблицы соответствий.
2. `publish` с `dryRun:true` на **одной** группе → проверить payload глазами.
3. `publish` с `dryRun:false` на **одной тестовой** группе → открыть в Alfa, проверить,
   что группа/расписание/состав легли верно; уточнить обязательные поля по ошибкам.
4. Массовая публикация остальных групп (кнопка на вкладке «Группы»).

## Клиент (ещё не подключён)
Когда справочники сверены — на вкладке «Группы» добавить кнопку «Опубликовать в Alfa»
с предпросмотром (что именно создастся) и защитой от повторной публикации уже
опубликованных групп (хранить `alfaGroupId` на группе в модели).
