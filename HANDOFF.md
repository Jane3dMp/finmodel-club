## Стек данных / синхронизация (СДЕЛАНО, работает)

- **Firebase Realtime Database** (проект `proznanie-finmodel`, регион `europe-west1`, план Spark/бесплатный) = **живой источник правды**, синхронизация в реальном времени между несколькими компами (цель — 5 одновременно). `FB_CONFIG` вшит в `index.html` (публичный, не секрет).
- **Модель данных в Firebase:** `scenarios/{имя}/grid/{lessonId}` — занятия по одному (гранулярно, у каждого занятия стабильный `id` → правки в разные занятия не затирают друг друга); `scenarios/{имя}/rest` — всё остальное одним JSON-блобом (иначе имена-ключи с точками, напр. курс «3D Unity 2.0», ломают запись в Firebase).
- **Google-таблица** (Apps Script, `S.gasUrl`) — только ручной резерв (кнопка «Выгрузить в таблицу» в модалке «Хранилище»). **localStorage** — локальный кэш.
- **Полностью автосохраняемая живая модель (как в Альфе, по просьбе Жанны):** каждое изменение само уходит в Firebase (`fbSchedulePush`, дебаунс 700 мс) и в localStorage; при обновлении страницы состояние восстанавливается. **Кнопка «Сохранить» убрана**, **всплывашки `beforeunload` больше нет** (на закрытии только тихий флаш отложенного `fbPush`), **Ctrl/⌘+S** перехвачен → тост «Всё сохраняется автоматически». Статус в шапке: «✓ Сохранено · онлайн» / «● Сохраняю…». Селектор версий из шапки убран — **переключение/создание сценариев осталось в ☰ Меню → «⋮ Версии модели»** (данные сценариев не тронуты).
- Ключевые функции: `fbStart`(auth-гейт) → `fbConnect` → `fbPush`/`fbSchedulePush`, `onRemoteModel`/`_applyRemote`, `_gridMap`/`_restStr`, `_ensureLessonIds`.

## ⚠️ КРИТИЧЕСКИЕ ФИКСЫ РАСПИСАНИЯ (июль 2026) — НЕ РЕГРЕССИРОВАТЬ

Жаловались: «рандом при копировании/сохранении, скачет, не могу сохранить, особенно с 2–3 компов». Две причины, обе починены и проверены в браузере:

1. **Занятия опознаём по СТАБИЛЬНОМУ `id`, НЕ по индексу массива.** `_applyRemote` пересобирает `S.grid=Object.values(remote.grid)` → индексы съезжают → клик/сохранение/перенос попадали не в то занятие. Теперь: рендер сетки кладёт `data-id` и вызывает `editLes(_lesIdx('id'))` (индекс считается в момент клика); `dragStart`/`dropLes` по `_dragId`+`_lesById`; модалка хранит `dataset.lesId`, все её действия — через `_modalLesIdx()` (id, фолбэк на idx) с гвардами от `idx=-1` (иначе `splice(-1,1)` удалял последнее занятие). Хелперы: `_lesById`/`_lesIdx`/`_modalLesIdx`. `_ensureLessonIds` теперь лечит и **дубли id** (старые копии). `dupLes` даёт копии **свой** `id` (`copy.id=_lid()`) — раньше копировал вместе с id → в Firebase (карта по id) копия затирала оригинал.
2. **Конкурентная синхронизация затирала свежую правку.** Пока карточка открыта, remote с другого компа откладывается в `_fbPending`; `closeLes` применял этот УСТАРЕВШИЙ снимок поверх → правку откатывало. Теперь `saveLes`/`delLes` ставят `_savedInModal=true`; `closeLes` применяет `_fbPending` **только на «Отмене»**, а после сохранения/удаления — отбрасывает (наша правка новее, чужие изменения подтянет живой слушатель). Плюс `saveLes`/`delLes` при `FB_ON` шлют `fbPush()` **сразу** (не через дебаунс) — закрывает окно гонки.

**Один и тот же состав у группы 2×/нед:** `saveLes` → `_syncGroupKids(l)` копирует состав сохранённого занятия во ВСЕ занятия той же группы (курс + номер группы; без номера не синхронизируем). Кнопка «🔗 Свести составы групп» (сетка → ⋯ Ещё, `syncAllGroups`) разом сводит все многозанятийные группы к объединению их составов (со снимком в Историю).
